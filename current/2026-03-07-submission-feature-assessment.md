# 本地编写与提交评测功能：综合评估（2026-03-07）

## 1. 目标与使用场景

目标：支持学生在本地写好代码后，在题目详情中点击“提交代码”，进入提交页面，粘贴代码并点击“提交评测”，将代码上传到 Matrix 服务器进行评测并返回结果。

本轮范围：仅评估、澄清需求、列出资料与 API 清单、给出落地开发思路；**不开始功能编码**。

---

## 2. 现状评估（基于当前仓库）

### 2.1 已具备能力

1. 登录态能力
   - 通过 Cookie 登录并持久化（`globalState`）。
   - 业务 API 调用已统一携带 Cookie。

2. 题目详情展示能力
   - 已有题目详情预览 Webview（`assignmentPreview`）。
   - 已支持 Markdown 渲染、附件展示、代码块高亮。

3. 服务层基础
   - 已有 `CourseService`，支持课程列表、题目列表、题目详情。
   - 已具备响应体解码能力：`aes-256-gcm` / `base64`。

4. 扩展入口与命令体系
   - 命令注册、TreeView、题目点击预览链路完整。

### 2.2 明显缺口

1. **提交链路完全缺失**
   - 无“提交代码”命令。
   - 无提交页面（输入代码、提交按钮、状态反馈）。
   - 无提交服务（提交请求、轮询评测结果、错误处理）。

2. **提交 API 未定义**
   - 仓库暂无任何提交/评测接口实现。
   - 暂不清楚提交请求是否需要加密编码、字段格式、语言 ID。

3. **交互机制待补齐**
   - 当前详情 Webview 以展示为主，提交流程需要可交互 Webview（消息通信）。

## 3. 必需资料与 API 清单（开发前必须拿到）

以下信息是“可开发提交功能”的最小前置条件。

### 3.1 接口文档（必须）

1. 提交代码接口
   - 方法与路径（例如：`POST /api/courses/{courseId}/assignments/{assignmentId}/submissions`）。
   - 请求体字段：代码、语言、文件名/多文件结构、附加参数。
   - 请求体编码方式：明文 JSON / base64 / aes-256-gcm。

2. 查询评测结果接口
   - 方法与路径（例如：`GET /.../submissions/{submissionId}`）。
   - 返回状态机：排队、编译中、评测中、完成、失败等。
   - 最终结果字段：通过情况、得分、耗时、内存、编译错误、运行错误、测试点详情。

3. 语言列表接口（若有）
   - 方法与路径（例如：`GET /api/languages` 或题目详情内返回可选语言）。
   - 语言 ID 与显示名映射。

4. 提交记录接口（可选但推荐）
   - 便于后续展示“最近提交结果”。

### 3.2 字段语义与业务规则（必须）

1. 提交限制规则
   - 是否有提交次数上限、提交间隔限制、截止后限制。

2. 权限规则
   - 学生/助教/教师角色的提交权限差异。

3. 错误码规范
   - 未登录、无权限、次数超限、题目关闭、代码过长、语言不支持等。

4. 评测结果规范
   - Verdict 枚举（AC/WA/TLE/MLE/RE/CE/PE 等）及中文映射。

### 3.3 联调材料（必须）

1. 一组可复现的接口样例
   - 成功提交 + 成功评测。
   - 编译失败。
   - 运行失败。
   - 权限错误/未登录。

2. 环境信息
   - 生产/测试环境 Base URL。
   - Cookie 的有效范围与过期策略。

## 4. 建议的 MVP 交互流程（以最小可用为目标）

1. 用户在题目详情页点击“提交代码”。
2. 打开“提交页面”Webview：
   - 题目信息简版（课程、题目、截止时间）。
   - 语言选择（下拉）。
   - 代码输入区域（粘贴）。
   - “提交评测”按钮。
3. 点击提交后：
   - 立即显示“提交中/排队中”。
   - 后台轮询评测状态。
4. 评测完成后：
   - 显示 Verdict、得分、耗时、错误信息（若有）。

## 5. 技术实现思路（分阶段）

### Phase A：数据与服务层

1. 新增 `SubmissionService`：
   - `submitCode(...)`
   - `fetchSubmissionResult(...)`
2. 统一复用现有请求/鉴权/解码策略。
3. 新增提交相关模型（SubmissionRequest / SubmissionResult / Verdict）。

### Phase B：命令与页面

1. 新增命令 `matrix-on-vscode.submitCode`。
2. 在题目详情页增加“提交代码”入口（按钮）。
3. 新增提交 Webview 页面并建立 message 通信。

### Phase C：评测状态与反馈

1. 提交后轮询状态（可配置间隔 + 超时）。
2. 状态映射与 UI 展示（队列中/评测中/完成）。
3. 失败场景处理（网络异常、登录过期、业务错误）。

### Phase D：质量保障

1. Service 层单测（接口解析、状态映射、错误处理）。
2. 关键交互手工回归（登录、提交、评测完成、失败）。

## 6. 需要你提供/确认的关键信息（优先级从高到低）

1. **提交接口与查询接口的真实路径和字段**（最高优先级）。
2. **语言 ID 与语言名称映射**。
3. **评测结果字段定义**（尤其是编译错误和测试点明细结构）。
4. **请求体是否需要加密编码**（与现有详情接口一致或不同）。
5. **提交限制规则**（次数、频率、截止后行为）。

## 7. 风险与应对

1. 风险：接口字段与预期不一致。
   - 应对：先做 API schema 适配层，避免 UI 直接依赖原始字段。

2. 风险：评测为异步且状态复杂。
   - 应对：定义统一状态机并集中在 Service 层映射。

3. 风险：登录态失效导致提交失败。
   - 应对：沿用现有“未登录 -> 引导重新登录”处理链路。

4. 风险：Webview 脚本交互增加复杂度。
   - 应对：先做最小消息协议（submit/requestStatus/result/error）。

---

## 8. 验收标准（MVP）

1. 在题目详情中可进入提交页面。
2. 粘贴代码后可提交到服务器。
3. 可看到评测中状态与最终结果。
4. 登录失效、参数错误、评测失败时有明确错误提示。
5. 不影响现有课程/题目浏览与预览功能。

## 10. 2026-03-16 实施更新（基于 client 前端代码对齐）

### 10.1 已确认的接口与请求格式

1. 提交代码
   - `POST /api/courses/{courseId}/assignments/{ca_id}/submissions`
   - 请求体（编程题）：
     - `detail.answers`: 数组，每项为 `{ name, code }`
     - `code` 采用 Base64 编码（前端对齐实现）

2. 查询提交历史
   - `GET /api/courses/{courseId}/assignments/{ca_id}/submissions`
   - 返回列表内可用 `sub_ca_id` / `submit_at` / `grade` 标识最新提交与评测状态

3. 查询单次提交详情
   - `GET /api/courses/{courseId}/assignments/{ca_id}/submissions/{sub_ca_id}`
   - 返回单次提交细节，包括 `answers`、`grade`、`report`

### 10.2 扩展侧已实现内容

1. 新增 `SubmissionService`
   - `submitCode(...)`
   - `fetchSubmissions(...)`
   - `fetchSubmissionDetail(...)`
   - `pollSubmissionResult(...)`

2. 新增提交 UI 与命令
   - `matrix-on-vscode.submitCode`
   - 题目节点右键可直接打开提交面板
   - 题目预览页新增“提交代码”按钮（command URI）

3. 新增提交 Webview 交互
   - 语言选择、文件名、代码输入
   - 提交状态日志（提交中、轮询中、完成/超时）
   - 结果展示（verdict、grade、submission id、报告摘要/原始 report）

4. 题目详情字段补齐

   - `gradeAtEnd`

### 10.3 剩余工作

1. 基于真实课程数据做端到端回归（AC/CE/RE/超时）。
2. 细化 verdict 映射（不同 report 结构分支）。
3. 增加提交服务的单测覆盖。
